# 当前基线审计

更新时间：2026-04-12

## 审计结论

当前仓库已经从“停更遗留项目”推进到了“可持续重构中的遗留系统”阶段。

这不是一句空话，原因很明确：

- 已有可重复的遗留运行时基线
- 已有持续可执行的 PHPUnit 回归
- 已有 GitHub Actions CI
- 安装主路径已经完成现代化切换
- 支付层已完成一轮大规模收口和通道退役
- 后台替换已经从评估走到了真实页面并行迁移

当前更准确的项目状态应理解为：

- 核心业务链：已有护栏，可继续重构
- 安装部署链：已基本脱离旧 SQL 安装路径
- 后台系统：处于新旧并行迁移期
- 升级准备：已清掉一批阻塞，但还没进入正式 Laravel 升级阶段

## 已完成进度

### 1. 遗留运行时与测试基线

已完成：

- 恢复 PHP 7.4 遗留运行时基线
- 固定 `scripts/php74` / `scripts/composer74`
- 本地快速拉站路径可复用
- GitHub Actions `CI` 已建立并在主线使用
- PHPUnit 主线回归已扩展到 `OK (221 tests, 1031 assertions)`

影响：

- 后续所有重构都有了自动回归护栏
- 不是“凭感觉改遗留系统”

### 2. 订单与支付主链拆分

已完成：

- 订单创建、履约、支付完成、通知已拆为独立服务
- 订单创建已切到 DTO / 应用服务模式
- 支付入口、支付回调、第三方 SDK 边界已明显收口
- `Stripe` 已升级到现代 SDK 基线
- 已退役 `Paysapi`、`Vpay`、`PayJS`

影响：

- 订单域已不再集中挤在一个超大服务里
- 支付层已具备后续继续淘汰和替换的基础

### 3. 去守护进程与部署模型收口

已完成：

- 订单过期不再依赖延迟 worker
- 通知与副作用默认改为同步优先、异步可选
- Dockerfile / compose / Debian 文档已移除项目对 `supervisord` / 常驻 `queue:work` 的硬依赖

影响：

- 项目部署和排障复杂度明显下降
- 后续容器化和环境统一成本更低

### 4. 安装与数据模型现代化

已完成：

- 主安装路径切到 `migrate + bootstrap seed + 显式管理员创建`
- `install.sql` 已退出安装主路径，并已从仓库主路径移除
- migration / seed 分层已建立
- bootstrap / sample / forbidden 默认数据策略已形成

影响：

- 新环境不再依赖整包 SQL 安装
- 仓库卫生、安全性和可重复部署性都提升了

### 5. 前台与品牌统一

## 当前不建议立即做的事

### 1. 不建议立刻整体替换后台

原因：

- 现在后台壳迁移是正确方向，但仍处于并行迁移期
- 直接一刀切替后台，风险会高于收益

### 2. 不建议立刻跳升 Laravel 大版本

原因：

- 当前依赖和后台壳虽然已好转，但升级前清障还没完全结束
- 现在最值钱的是继续降低升级阻塞，而不是急着冲版本号

## 当前主线判断

如果把整个项目分成“是否值得继续投入”的几个等级：

- 已脱离“只能救火”
- 已进入“结构化治理”
- 还没进入“现代化收尾”

最准确的位置是：

- “结构化治理后期，后台替换中期，升级前清障前期”
