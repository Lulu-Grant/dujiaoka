# Dujiaoka 大整改执行方案

## 目标

本方案用于把现代化改造固定成“长周期、低打扰、按阶段持续推进”的执行模式。

最终目标不是只把旧项目修到能跑，而是把它推进成一套：

- 可持续维护
- 不依赖守护进程
- 有核心业务测试护栏
- 支付层可替换
- 后台可逐步脱离 Dcat
- 为 PHP / Laravel 升级做好准备

## 默认执行规则

- 默认由 Codex 连续推进，不在每个小节点停下来征求确认。
- 只有在出现高风险分叉、不可逆操作、外部真实信息缺失时才暂停。
- 每个稳定节点必须：
  - 跑当前回归测试
  - 更新 [refactor-upgrade-log.md](/Users/apple/Documents/dujiaoshuka/docs/refactor-upgrade-log.md)
  - 尽量同步 GitHub `master`
- 每个阶段优先处理“能降低后续返工成本”的任务，而不是只追求可见改动数量。

## 常驻基线监控子代理规则

为了避免主线程在连续推进时偏离当前基线，同时又不把开发节奏变成“每一步都要人工确认”，后续默认挂一个常驻基线监控子代理。

职责：

- 只负责检查是否严重偏离当前基线
- 不直接主导功能实现
- 不替代主线程做取舍
- 只在主线程遇到“是否继续下一步”的判断点时，给出轻量建议

检查范围：

1. 是否偏离当前总纲与当前执行批次
2. 是否引入高风险但未补护栏的改动
3. 是否出现“测试绿但结构方向明显跑偏”的情况
4. 是否出现文档、路由、资源注册表与真实代码状态失配
5. 是否在没有必要的情况下过早跳到升级、替换或安全专项之外的分支

严重偏离的判定标准：

- 改动明显脱离“后台壳扩容 + 旧兼容层收缩”主线
- 引入副作用风险，但没有同步补测试或边界说明
- 改动破坏统一模式，例如绕开资源注册表、绕开页面服务、重新把逻辑塞回旧兼容层
- 同一资源出现两套长期并存且没有明确淘汰方向的实现

子代理输出规则：

- 默认不阻塞主线程继续推进
- 若未严重偏离，只返回“可继续”
- 若存在风险，只给出“建议暂停检查”的提示和理由
- 建议不是强制中断，最终仍由主线程决定

主线程协作规则：

- 主线程继续直接推进开发
- 在遇到“是否继续下一步”“是否扩这条线”“是否切去别的阶段”这类节点时，优先让常驻子代理先做一次偏航检查
- 如果子代理判断未严重偏离，则继续推进，不额外等待人工确认

实际运行协议：

1. 主线程完成一个稳定节点后，先总结本轮改动
2. 当出现“是否进行下一步”的判断点时，优先让常驻子代理做一次 10 秒级偏航检查
3. 检查顺序固定为：
   - 是否仍属于当前批次：后台壳扩容、旧兼容层收缩、支付收口、升级前清障
   - 是否把逻辑写回旧 Dcat 或新增长期并存的平行实现
   - 是否引入高风险改动但没有同步测试、烟雾或日志
   - 是否提前跳去大规模升级、重写或完全新战线
   - 是否破坏当前文档口径、路由/资源注册表或稳定节点记录
4. 若子代理返回“可继续”，则主线程默认继续推进，不再额外等待人工确认
5. 若子代理返回“建议暂停检查”，则主线程记录风险理由并自行决定是否继续；该建议不是强制中断

## 六大阶段

### 阶段 0：现状固化

目标：

- 确保已经完成的运行时、测试、安装、主题和文档基线始终可复用

退出标准：

- 主线持续可测试
- 文档与仓库状态一致
- 不需要回头补基础设施

### 阶段 1：后台替换准备与并行迁移

目标：

- 将现有旧后台兼容层继续压缩成最小接线层
- 把新后台壳做成可批量迁移的底座

任务：

1. 扩大后台壳资源覆盖面
2. 将旧 Dcat 中的表单保存、批量动作、导入导出继续抽成普通服务
3. 为配置型和操作型页面建立统一的壳页规范
4. 将后台迁移按批次推进，而不是零散替换

退出标准：

- 新后台壳承接一批真实高频后台页面
- 旧 Dcat 仅剩少数复杂页
- 后台替换进入可持续并行迁移阶段

### 阶段 2：支付层现代化收口

目标：

- 将支付层整理成“可保留、可退役、可替换”的稳定结构

任务：

1. 明确保留通道、观察通道、退役通道
2. 完成 PayPal 剩余退场准备
3. 继续稳住 Stripe 的现代 SDK 基线
4. 持续补强支付幂等、签名失败、重复通知等测试

退出标准：

- 支付层生命周期清楚
- 历史通道不再污染默认样例与主路径
- 升级不会被旧支付 SDK 再次反向拖住

### 阶段 3：安装、数据模型与配置现代化

目标：

- 彻底摆脱整包 SQL 安装模型

任务：

1. 补齐 migration / seed 覆盖
2. 清理安装器残余假设
3. 明确 bootstrap、sample、forbidden 三类默认数据
4. 将系统设置逐步转向更长期可维护的配置域

退出标准：

- `install.sql` 只剩历史参考意义
- 新安装完全依赖现代路径
- 环境、种子和管理员创建可重复

### 阶段 4：安全治理专项

目标：

- 专项收掉默认项、后台、支付回调和仓库卫生方面的历史风险

任务：

1. 默认入口与默认项治理
2. 支付签名、金额、幂等与重放保护专项检查
3. 后台权限、上传、模板输入、运行时配置专项检查
4. 凭据、密钥、CI 与仓库卫生清理

退出标准：

- 有明确的安全状态清单
- 每项风险有处理状态，而不是零散记忆

### 阶段 5：升级前清障

目标：

- 为 PHP / Laravel 升级创造干净切口，而不是直接跳版本

任务：

1. 清剩余依赖阻塞
2. 降低旧 Laravel / 旧 Dcat 绑定面
3. 设计现实的分步升级路径
4. 让后台替换与升级路线并轨

退出标准：

- 有现实可执行的升级路径
- 不存在“理论可升、实操升不动”的断层

### 阶段 6：现代化收口

目标：

- 从“遗留系统治理中”切换到“现代维护项目”

任务：

1. 新后台壳成为主承载
2. 支付层生命周期稳定
3. 安装链与部署链统一
4. 输出现代化文档、运维手册和架构说明

退出标准：

- 项目已具备长期维护形态

## 批次执行规则

每个批次内遵循：

1. 选择一个明确整改点
2. 实施代码或文档变更
3. 补测试或验证
4. 更新升级日志
5. 在稳定节点同步 GitHub `master`

这意味着后续推进默认会以“阶段任务包”的形式连续执行，而不是每做一个小步骤就等待确认。
