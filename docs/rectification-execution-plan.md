# Dujiaoka 大整改执行方案

## 目标

本方案用于把当前现代化改造从“逐点推进”切换为“按阶段连续执行”。

执行原则：

- 默认由 Codex 按阶段连续推进，不在每个小节点停下来征求确认。
- 只有在遇到高风险分叉、业务取舍明显冲突、或需要真实生产信息时才暂停。
- 每完成一个重要节点，必须同步更新：
  - [refactor-upgrade-log.md](/Users/apple/Documents/dujiaoshuka/docs/refactor-upgrade-log.md)
- 每形成一个稳定检查点，应运行当前回归测试，并尽量同步到 GitHub `master`。

## 执行阶段

### 阶段 A：支付层整治收口

目标：

- 先把最脆弱、最分散的支付控制器收敛到统一编排模型

任务：

1. 继续将剩余通知型支付控制器接入统一回调骨架
2. 提取第三方 SDK 型回调的包装服务，减少控制器直接依赖 facade / 原始 webhook
3. 梳理所有 `pay_handleroute` 历史不一致问题
4. 产出支付网关清单，标注：
   - 已接入统一骨架
   - 仍为旧式实现
   - 依赖过时 SDK
   - 建议淘汰

退出标准：

- 主流通知型支付控制器不再重复手写订单解析与完成支付入口
- 支付层整改状态有清单可查

### 阶段 B：订单域编排清理

目标：

- 让控制器、应用服务、领域服务边界进一步清晰

任务：

1. 继续压缩 Home 控制器和支付入口控制器
2. 清理剩余大方法和共享可变状态
3. 明确订单状态流转规则和异常路径
4. 评估是否引入更明确的领域目录结构

退出标准：

- `OrderProcessService` 只承担薄编排职责
- 关键流程的输入输出边界清楚

### 阶段 C：安装与数据模型现代化

目标：

- 摆脱 `install.sql` + 安装器写 `.env` 的旧模型

任务：

1. 盘点 `install.sql` 对应的数据域
2. 先拆 schema migration，再拆 seed data
3. 识别安装器中的环境写入、默认管理员、锁文件逻辑
4. 规划新的 bootstrap 路径

退出标准：

- 安装层有迁移方案和拆解顺序
- 不再依赖整包 SQL 作为未来唯一安装方式

### 阶段 D：安全治理与敏感面收口

目标：

- 清掉最危险的历史遗留安全问题

任务：

1. 默认凭据与默认后台入口治理
2. 回调签名、重放、金额校验专项清单
3. 外部通知 / webhook 出口治理
4. 文档化密钥轮换与仓库卫生规则

退出标准：

- 高风险默认项有清单、有处理状态、有替代方案

### 阶段 E：升级前清障

目标：

- 为 Laravel / PHP 升级创造条件

任务：

1. 标注升级阻塞包和替代路线
2. 继续减少 Laravel 6 时代写法耦合
3. 评估 Dcat Admin 阻塞程度
4. 制定分步升级路径

退出标准：

- 有可执行的升级前置清单，而不是“直接跳版本”

当前进展：

- 安装现代化、测试护栏、品牌统一等前置准备已基本收口
- 已开始建立升级前清障清单与后台替换评估资料

## 同步规则

每个阶段内遵循：

1. 选择一个明确整改点
2. 实施代码或文档变更
3. 补测试或验证
4. 更新升级日志
5. 在稳定节点同步 GitHub `master`

这意味着后续推进默认会以“阶段任务包”的形式连续执行，而不是每做一个小步骤就等待确认。
